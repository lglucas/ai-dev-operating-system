# 2026-05-04 — v0.4.4 repo hardening (branch protection + template + WIZARD Stage 0.5)

## Trigger

User reported: "Ontem eu dei início a um novo projeto, e clonei este nosso repositório, como ele deve ser (afinal, é público, foi feito pras pessoas usarem), e logo após os primeiros passos do wizard, ele já fez um primeiro commit no NOSSO repositório daqui."

The OS repo had no protection against derived projects accidentally pushing into it. README and `docs/installation.md` instructed users to `git clone` directly, leaving `origin` pointing to the OS repo.

## Decisions

### 1. Three-layer hardening, in priority order

| Layer | Where | Why |
|---|---|---|
| 1. Branch protection on `main` | GitHub server-side | Prevents direct push to `main` even if origin is misconfigured. Standard open-source posture. |
| 2. GitHub Template flag | GitHub server-side | Gives users the "Use this template" button → fresh repo, no shared history with OS. Standard for boilerplate repos (Vercel, shadcn, etc.). |
| 3. WIZARD Stage 0.5 | In-repo | Educational + executable defense for users who clone (fallback path). |

User asked whether the 3 layers are common practice. Confirmed: they are the canonical pattern for public template repos. We're not inventing — we're catching up.

### 2. Reordered priority based on user follow-up questions

User asked 3 follow-up clarifications:
- "Is this market practice?" → yes, examples: Vercel, shadcn, Cloudflare templates, all `create-*` CLIs.
- "How would someone collaborate?" → fork → branch → PR. We had no `CONTRIBUTING.md` describing this. Added.
- "Do the 3 layers teach a vibe coder how to set up Git?" → no, that's the gap. Stage 0.5 was rewritten to be explicitly educational, not just executional. Workshop deferred.

### 3. `.aios-self` marker — detection logic refined mid-implementation

Initial design: `.aios-self` marker auto-skips Stage 0.5 / detach script when present.

Bug found during implementation: the marker travels with `git clone` AND with "Use this template", so a derived project would see `.aios-self` and Stage 0.5 would skip — defeating the entire purpose.

Fix: detection requires BOTH `.aios-self` AND `origin` URL containing `lglucas/ai-dev-operating-system` AND user confirmation. The detach scripts now show a confirmation prompt instead of auto-aborting. The detach action itself removes `.aios-self` as cleanup, so derived projects shed the marker on first detach.

Edge case acknowledged in RELEASE-NOTES: if a user uses "Use this template" but renames the new repo to `ai-dev-operating-system` AND keeps `.aios-self`, detection becomes ambiguous. User-resolved via Stage 0.5 prompt.

### 4. Walking the talk

Implementation done on a feature branch (`feat/repo-hardening-detach`) and merged via PR — not direct push to main. The first PR after enabling branch protection is the PR that documented and enabled it.

## Versioning

Bumped to v0.4.4 (patch within the v0.4.x quick-wins line). v0.5 architectural decisions remain deferred (trust tiers, plugin manifest, parallel sprints, allowlists, registry-of-registries) — those are bigger and need separate user decisions.

## Workshop deferral

User requested a workshop covering: GitHub account creation, Git basics, `.env` discipline, end-to-end WIZARD walkthrough. Agreed to defer until v0.4.4 ships and we can iterate on what's actually confusing about the workflow.
